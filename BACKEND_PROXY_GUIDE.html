1. 문서 목적

이 문서는 일반적인 백엔드 설명이 아니라 현재 저장소의 실제 코드 구조를 기준으로 작성되었습니다. 구현 담당자는 이 문서만으로 변경 이유, 대상 파일, API 형태, 서버 역할과 검증 순서를 파악할 수 있어야 합니다.

보호할 것

OpenAI API 키, 사용자 사진, 일기 내용, 서비스 사용량과 비용입니다.

유지할 것

upload → write → preview 화면 흐름, 로컬 Mock 모드, 훅과 UI의 공개 인터페이스는 최대한 유지합니다.

바꿀 것

브라우저에서 OpenAI로 직접 보내는 요청을 자체 백엔드 API 호출로 교체합니다.

HTML과 CSS의 역할 이 파일은 HTML 문법과 내부 CSS로 작성된 설명 문서입니다. 실제 프록시는 HTML/CSS가 아니라 서버에서 실행되는 JavaScript 또는 TypeScript로 구현해야 합니다.

2. 백엔드 프록시의 정의

이 프로젝트에서 백엔드 프록시란 앱과 OpenAI API 사이에 위치하여, 프론트엔드의 요청을 검증한 뒤 서버에만 보관된 비밀 키로 OpenAI를 대신 호출하고, 검증된 결과만 앱에 돌려주는 서버 계층을 의미합니다.

프로젝트에서 프록시가 담당해야 하는 일

  • OpenAI API 키를 서버 환경변수로만 보관합니다.
  • 사진 형식, 이미지 크기, 제목과 본문 길이를 다시 검증합니다.
  • 일기 분석 요청과 색연필 변환 요청을 OpenAI에 전달합니다.
  • OpenAI 원본 응답을 현재 앱의 타입에 맞게 정규화합니다.
  • 타임아웃, 요청 제한, 비용 제한과 오류 변환을 적용합니다.
  • 사진과 일기 원문을 서버 로그에 남기지 않습니다.
  • 사용자가 원격 처리에 동의했는지 확인할 수 있는 구조를 제공합니다.
프록시는 단순 전달 주소가 아닙니다. 요청을 아무 검증 없이 OpenAI로 전달하면 공격자가 프록시를 이용해 서버의 API 비용을 소비할 수 있습니다. 입력 검증, 인증 또는 호출 제한이 반드시 함께 있어야 합니다.

6. 프론트엔드와 백엔드의 API 계약

6.1 일기 분석

주소: POST /api/diary/analyze

용도: 사진과 일기를 함께 분석하여 키워드, 감정, 첨삭 대상과 한줄평을 생성합니다.

요청 JSON 
{
  "photoDataUrl": "data:image/jpeg;base64,...",
  "title": "바다에 간 날",
  "content": "가족과 바다에 가서 모래성을 만들었다.",
  "weather": "sunny"
}
성공 응답 JSON 
{
  "photoKeywords": ["바다", "가족", "모래성"],
  "diaryKeywords": ["여행", "놀이"],
  "emotions": ["즐거움", "행복"],
  "highlightWords": ["모래성"],
  "highlightSentence": "가족과 바다에 가서 모래성을 만들었다.",
  "comment": "가족과 함께한 즐거운 마음이 잘 느껴져요."
}

서버 검증 규칙

  • photoDataUrl은 허용된 이미지 형식만 받습니다.
  • title은 공백만으로 구성될 수 없고 최대 30자입니다.
  • content는 공백을 제외한 길이가 최소 20자이고 최대 55자입니다.
  • weather는 프로젝트의 다섯 가지 날씨 값 중 하나여야 합니다.
  • 첨삭 단어와 문장은 실제 본문에 포함된 경우에만 응답에 유지합니다.
  • comment는 존댓말 한 문장으로 작성하고, 공백을 포함해 최대 50자로 제한합니다.

6.2 색연필 그림 변환

주소: POST /api/diary/sketch

용도: 업로드한 사진을 현재 프로젝트의 색연필 그림 프롬프트로 변환합니다.

형식: multipart/form-data를 권장합니다.

프론트 요청 개념 
const formData = new FormData();
formData.append("photo", photoBlob, "photo.jpg");

await fetch(`${API_BASE_URL}/api/diary/sketch`, {
  method: "POST",
  body: formData,
});
성공 응답 JSON 
{
  "sketchDataUrl": "data:image/jpeg;base64,..."
}

서버 검증 규칙

  • 한 요청에 사진 한 장만 허용합니다.
  • JPG, PNG, WEBP만 허용합니다.
  • 프로젝트 업로드 정책에 맞춰 최대 10MB로 제한합니다.
  • 파일 확장자가 아니라 실제 MIME과 이미지 디코딩 가능 여부를 확인합니다.
  • OpenAI에 보내기 전에 필요한 경우 크기를 다시 축소합니다.

6.3 공통 오류 응답

오류 JSON 
{
  "code": "RATE_LIMITED",
  "message": "요청이 많아요. 잠시 후 다시 시도해 주세요.",
  "retryable": true
}

7. 백엔드 서버 설계

백엔드는 Node.js와 TypeScript로 구성하는 것이 현재 프로젝트와 언어를 통일하기 쉽습니다. Express, 서버리스 함수 또는 Edge Function 중 어느 실행 환경을 선택하더라도 아래 책임은 동일합니다.

라우트 계층

HTTP 요청을 받고 입력을 검증한 뒤 서비스 계층을 호출합니다.

analyzeDiary.ts, createSketch.ts

OpenAI 서비스 계층

서버 비밀 키, 모델, 프롬프트와 OpenAI 오류 변환을 담당합니다.

services/openai.ts

검증 계층

문자열 길이, 날씨 값, 이미지 MIME 및 용량을 검사합니다.

validation/diary.ts

보호 계층

요청 제한, 인증, CORS, 본문 크기 제한과 공통 오류 처리를 담당합니다.

middleware/

서버 환경변수

diary-proxy-server/.env.example 
OPENAI_API_KEY=
OPENAI_ANALYSIS_MODEL=gpt-4o-mini
OPENAI_IMAGE_MODEL=gpt-image-1
OPENAI_IMAGE_QUALITY=medium
ALLOWED_ORIGIN=https://summer-vacation-diary.web.tossmini.com
REQUESTS_PER_MINUTE=10
PORT=3000
서버 비밀 환경변수에는 VITE_ 접두어를 사용하지 않습니다. OPENAI_API_KEY는 서버의 비밀 저장소에만 설정하고, 실제 값이 들어 있는 .env 파일은 Git에 커밋하지 않습니다.

분석 라우트의 처리 순서

  1. 요청 본문 크기를 제한합니다.
  2. 사용자 또는 IP 단위 요청 횟수를 확인합니다.
  3. 사진, 제목, 본문과 날씨를 검증합니다.
  4. 서버가 OpenAI 요청 메시지와 프롬프트를 생성합니다.
  5. 30초 타임아웃과 함께 OpenAI를 호출합니다.
  6. 응답 JSON을 파싱하고 필드별 타입 및 최대 개수를 검증합니다.
  7. comment가 공백 포함 50자를 넘으면 문자열을 임의로 자르지 않고 잘못된 응답으로 처리합니다.
  8. 본문에 존재하지 않는 첨삭 단어와 문장을 제거합니다.
  9. 검증된 DiaryAnalysis만 프론트엔드에 반환합니다.

이미지 변환 라우트의 처리 순서

  1. 멀티파트 업로드에서 사진 한 장만 받습니다.
  2. 파일 크기와 실제 이미지 형식을 검증합니다.
  3. 필요하면 해상도와 품질을 서버 기준으로 축소합니다.
  4. 현재 styleTransfer.ts의 색연필 프롬프트를 서버로 이동합니다.
  5. 120초 타임아웃과 함께 OpenAI Images API를 호출합니다.
  6. 결과 이미지를 JPEG로 재압축하고 앱이 저장할 수 있는 크기로 제한합니다.
  7. Base64 데이터 URL 또는 임시 이미지 URL을 반환합니다.

8.5 기존 Mock 모드 유지

API 키의 존재 여부 대신 VITE_USE_MOCK 또는 백엔드 주소의 존재 여부로 로컬 모드를 선택합니다. Mock을 명시적으로 켰거나 백엔드 주소가 비어 있으면 원격 요청을 보내지 않습니다. 이 판단은 분석과 스케치 서비스가 서로 다르게 계산하지 않도록 공통 실행 설정에 둡니다.

src/config/runtime.ts 
const API_BASE_URL =
  (import.meta.env.VITE_API_BASE_URL ?? "").trim();
export const isUsingMock =
  import.meta.env.VITE_USE_MOCK === "true" || API_BASE_URL === "";
export { API_BASE_URL };

src/services/diaryAnalysis.ts와 src/services/styleTransfer.ts는 공통 isUsingMock 값과 사용자 동의를 함께 확인합니다. Mock 모드이거나 사용자가 원격 처리에 동의하지 않았다면 각각 로컬 분석과 로컬 필터를 사용합니다. src/components/PreviewStep.tsx의 체험 모드 안내도 기존 isAiConnected가 아니라 isUsingMock을 기준으로 표시합니다.

두 서비스의 공급자 선택 개념 
const useLocalProcessing =
  isUsingMock || !remoteProcessingConsent;

const analysisPromise = useLocalProcessing
  ? analyzeWithMock(input)
  : analyzeWithProxy(input);

const sketchPromise = useLocalProcessing
  ? sketchWithLocalFilter(photoDataUrl)
  : sketchWithProxy(photoDataUrl);

8.6 한줄평 표시 방식 변경

선생님 한줄평은 미리보기와 최종 저장 이미지에서 최대 2줄로 표시합니다. 표시 영역을 초과하면 마지막 줄에 말줄임표를 적용합니다.

  • src/App.css에서 한줄평의 -webkit-line-clamp 값을 1에서 2로 변경합니다.
  • src/utils/diaryImage.ts에서 Canvas의 텍스트 너비를 측정해 한줄평을 최대 2줄로 나누어 출력합니다.
  • 미리보기와 저장 이미지에 동일한 줄 수와 말줄임표 규칙을 적용합니다.
UI 변경을 최소화할 수 있는 이유 기존 훅은 analyzeDiary()와 transferPhotoToSketch()만 호출합니다. 이 함수들의 반환 타입을 유지한 채 내부 공급자만 프록시로 교체하면 화면 컴포넌트의 변경 범위를 줄일 수 있습니다.

9. 개인정보 처리와 사용자 동의

이 서비스는 가족 또는 아이의 얼굴, 위치, 주변 환경이 포함된 사진을 처리할 가능성이 높습니다. 따라서 기술적인 프록시 도입과 별도로 사용자에게 원격 처리 사실을 명확히 안내해야 합니다.

권장 화면 위치

src/components/PhotoUploadStep.tsx의 사진 선택 영역 아래 또는 사용자가 처음으로 “일기 쓰러 가기”를 누르는 시점에 안내를 표시합니다.

사용자 안내 문구 예시

색연필 그림과 선생님 한줄평을 만들기 위해 선택한 사진과 작성한 일기 내용이 분석 서버로 전송됩니다. 전송된 정보는 결과 생성 목적으로만 사용하며, 결과를 반환한 뒤 서버에 별도로 저장하지 않습니다. 사용하는 배포 환경에서 임시 파일이 생성되는 경우에는 보관 시간과 삭제 방식을 개인정보 처리 안내에 별도로 명시합니다.

권장 선택지

  • 원격 분석 사용: 백엔드 프록시를 통해 분석과 이미지 변환을 실행합니다.
  • 로컬 체험 모드: 사진은 기기 안에서 필터 처리하고 한줄평은 Mock 결과를 사용합니다.
동의 상태와 서비스 실행 시점 현재 색연필 변환은 사용자가 업로드 화면을 벗어나는 순간 시작됩니다. 따라서 동의 확인은 반드시 그보다 먼저 끝나야 하며, 동의하지 않은 사용자의 사진을 원격 서버에 먼저 보내서는 안 됩니다.

보호 항목 필수 조치 이유
API 키 서버 비밀 환경변수로만 저장 프론트 번들 및 Git 노출 방지

CORS는 인증이 아닙니다. CORS는 브라우저의 교차 출처 요청을 제한할 뿐, 공격자가 서버에서 직접 API를 호출하는 것을 막지 못합니다. 가능하면 Toss 사용자 확인 또는 서버가 발급한 짧은 수명의 토큰을 사용하고, 최소한 강한 요청 제한을 적용해야 합니다.

11. 오류 처리와 기존 UX 연결

현재 앱에는 분석과 스케치의 로딩·성공·오류 상태, 명시적 재시도, 원본 사진 폴백이 이미 구현되어 있습니다. 프록시는 기존 상태 모델이 이해할 수 있는 오류 코드로 응답해야 합니다.

백엔드 상황 프론트 처리 사용자 경험

이미지 변환 실패 명시적 재시도 또는 원본 사진 사용 그림일기 완성은 계속 가능

12. 권장 구현 순서

  1. API 계약 고정

    현재 DiaryAnalysisInput, DiaryAnalysis 및 스케치 반환 타입을 기준으로 요청·응답 JSON을 먼저 확정합니다.

  2. 백엔드 기본 프로젝트 생성

    diary-proxy-server/를 만들고 TypeScript, 환경변수 로딩, CORS, JSON 크기 제한과 공통 오류 응답을 구성합니다.

  3. 일기 분석 API 이전

    diaryAnalysis.ts의 OpenAI 프롬프트, 요청 및 응답 파서를 백엔드로 이동하고 /api/diary/analyze를 구현합니다.

  4. 색연필 변환 API 이전

    styleTransfer.ts의 모델, 품질, 프롬프트와 OpenAI 요청을 백엔드로 이동하고 /api/diary/sketch를 구현합니다.

  5. 프론트 서비스 교체

    기존 공개 함수와 상태 타입을 유지하면서 내부 요청 주소만 자체 프록시로 변경합니다.

  6. 동의 및 로컬 모드 연결

    원격 전송 전에 사용자의 선택을 확인하고, 미동의 또는 개발 환경에서는 기존 Mock 분석과 로컬 필터를 사용합니다.

  7. 보호 장치 추가

    호출 제한, 요청 크기 제한, 이미지 검증, 로그 마스킹, 타임아웃과 운영 CORS를 적용합니다.

  8. 실기기 검증 및 배포

    일반 브라우저와 Toss WebView에서 분석, 변환, 재시도, 원본 폴백과 최종 저장을 확인한 뒤 운영 주소로 전환합니다.

13. 테스트 계획

정상 흐름

  • 사진과 20~55자 일기가 정상 분석되는가
  • 첨삭 단어와 문장이 본문에 실제로 존재하는가
  • 한줄평이 공백 포함 50자 이내로 반환되는가
  • 한줄평이 미리보기와 저장 이미지에서 동일하게 최대 2줄로 표시되는가
  • 색연필 이미지가 프론트 저장 제한에 맞는가
  • 미리보기와 최종 저장 이미지가 정상인가
  • Mock 모드가 백엔드 없이 동작하는가
  • 원격 처리에 동의하지 않으면 분석 서버를 호출하지 않는가

실패 흐름

  • 잘못된 날씨와 빈 제목을 서버가 거부하는가
  • 10MB 초과 사진을 거부하는가
  • 이미지로 위장한 파일을 거부하는가
  • 50자를 초과한 한줄평 응답을 잘못된 응답으로 처리하는가
  • OpenAI 타임아웃이 사용자용 오류로 변환되는가
  • 429와 콘텐츠 차단을 서로 다르게 처리하는가
  • 오류 응답에 내부 스택이나 API 키가 없는가

보안 확인

  • 빌드된 프론트 파일에서 sk- 형식의 키가 검색되지 않아야 합니다.
  • 브라우저 개발자 도구의 요청 헤더에 OpenAI 키가 없어야 합니다.
  • 서버 로그에 Base64 사진과 일기 원문이 없어야 합니다.
  • 짧은 시간의 반복 요청이 429로 제한되어야 합니다.
  • 허용하지 않은 브라우저 출처가 CORS 정책으로 차단되어야 합니다.

14. 배포 구성

백엔드는 Node.js 서버, 서버리스 함수 또는 Edge Function에 배포할 수 있습니다. 특정 플랫폼을 선택하기 전에도 다음 배포 원칙은 동일합니다.

백엔드 배포

  • 서버 비밀 저장소에 OPENAI_API_KEY 등록
  • 운영 프론트 주소를 ALLOWED_ORIGIN으로 등록
  • HTTPS 적용
  • 요청·오류·비용 모니터링 활성화
  • 사진과 일기 원문 로그 비활성화

프론트 배포

  • VITE_OPENAI_API_KEY 완전 제거
  • VITE_API_BASE_URL에 운영 프록시 주소 등록
  • 개발·스테이징·운영 주소 분리
  • 실제 Toss WebView에서 네트워크와 저장 확인
  • 개인정보 안내 문구 확인
도메인 연결 시 확인할 점 Apps in Toss의 실제 운영 출처가 확정되면 백엔드 CORS 설정을 해당 주소와 정확히 맞춰야 합니다. 임시로 모든 출처를 허용하는 * 설정을 운영 환경에 남기지 않습니다.

15. 적용 완료 기준

다음 조건을 모두 만족하면 백엔드 프록시 이전이 완료된 것으로 봅니다.

  • 프론트엔드 코드와 빌드 결과물에 OpenAI API 키가 없습니다.
  • 일기 분석은 /api/diary/analyze를 통해 수행됩니다.
  • 색연필 변환은 /api/diary/sketch를 통해 수행됩니다.
  • 백엔드가 모든 입력을 다시 검증합니다.
  • 요청 제한, 타임아웃, 오류 정규화와 로그 마스킹이 적용됩니다.
  • 원격 전송 전에 사용자 안내 또는 동의 흐름이 실행됩니다.
  • 동의하지 않거나 백엔드가 없는 환경에서는 로컬 모드가 동작합니다.
  • OpenAI 오류가 발생해도 사용자의 사진과 작성한 일기는 유지됩니다.
  • 일반 브라우저와 Toss 실기기에서 전체 흐름을 검증했습니다.
기대 결과 기존 그림일기 UX는 유지하면서 API 키 노출, 무제한 비용 사용, 사용자 사진의 통제되지 않은 외부 전송 위험을 크게 줄일 수 있습니다. 또한 모델과 프롬프트, 제한 정책을 앱 재배포 없이 서버에서 관리할 수 있습니다.

이 문서는 Summer Vacation Diary 저장소의 현재 구조를 기준으로 작성한 설계 가이드입니다. 실제 구현 전에는 선택한 백엔드 배포 환경의 환경변수, 요청 크기, 실행 시간 및 CORS 제한을 함께 확인해야 합니다.
